# Ares Docs - Agent Instructions

## Package Manager

- Always use **Bun** as the package manager and runtime
- Run scripts with: `bun run <script-name>`
- Execute files with: `bun <file-path>`

## Project Context

This is a **documentation generation pipeline** for the Ares Car Rental graduation project. It:

- Packs repository source code via Repomix
- Sends context to an AI model to generate academic chapters
- Aggregates multi-part chapters into final Markdown files
- Renders all chapters into a single PDF via Quarto + MiKTeX (LuaLaTeX)

## Key Commands

| Command | Purpose |
|---------|---------|
| `bun run generate --all` | Generate all chapters via AI |
| `bun run generate --chapter 3` | Generate a single chapter |
| `bun run generate --dry-run` | Pack context and count tokens only |
| `bun run pdf` | Render generated Markdown → PDF |
| `bun run pdf:count` | Report accurate page count from existing PDF |
| `bun run pdf:clean` | Clean rebuild of PDF |
| `bun run typecheck` | TypeScript check (uses tsgo) |
| `bun run lint` | ESLint |
| `bun run format:check` | Prettier formatting check |

## Code Quality Checks

**Run in this order after any code changes:**

1. `bun run typecheck` — must pass (pre-existing errors in `validate-mermaid.ts` are known)
2. `bun run lint` — must pass
3. `bun run format:check` — must pass

## PDF Generation Architecture

- **`_quarto.yml`** — Quarto book config. Defines chapter order, PDF format (A4, Times New Roman 12pt, 1.5 line spacing, university-specified margins).
- **`pdf.ts`** — Bun script that:
  - Ensures MiKTeX is on PATH (`C:\Users\PC\AppData\Local\Programs\MiKTeX\miktex\bin\x64`)
  - Runs `quarto render --to pdf`
  - Parses the generated PDF for accurate page count
  - Accepts `--count-only`, `--clean` flags
- **`index.ts`** — After generation, checks for `_pdf/ares-docs.pdf` to report accurate pages; falls back to char-based estimation if PDF is absent.

## Rules

### File Operations

- Use `import { existsSync, readFileSync } from "node:fs"` for synchronous checks in non-async paths
- Use `Bun.file(path).text()` for async file reads in generation flow
- Use `Bun.write(path, content)` for file writes
- Generated chapters go into `generated/` directory (path configured via `OUTPUT_DIR` in `.env`)

### Quarto/PDF Rules

- **Never modify `_pdf/` contents manually** — it is fully generated by Quarto
- **Never commit `_pdf/` or `_freeze/`** — both are gitignored
- If modifying `_quarto.yml`, test with `bun run pdf` to verify the PDF still renders
- Missing LaTeX packages must be installed via MiKTeX CLI: `miktex packages install <name>`
- The MiKTeX auto-install setting may be off by default. If `quarto render` fails with "missing packages", install them manually or enable auto-install in MiKTeX Console.

### Environment Configuration

- **Never commit `.env`** — it contains API keys
- Copy `.env.example` to `.env` and fill in credentials
- Key env vars: `CUSTOM_API_ENDPOINT`, `CUSTOM_API_KEY`, `AI_MODEL`, `MAX_INPUT_TOKENS`, `MAX_OUTPUT_TOKENS`

### Generation Flow

1. Pack repository context (per-chapter include patterns)
2. Split large chapters into parts (token-based)
3. Stream AI generation per part
4. Aggregate parts → final chapter `.md`
5. (Separate step) Run `bun run pdf` to render all chapters to PDF

## TypeScript Style

- No `any` types
- No non-null assertions (`!`)
- All functions must have return types
- Use `zod` for runtime validation (see `lib/config.ts`)

## Console / Logging

- **Console methods are allowed** (this is a CLI tool, not a browser app)
- Prefer the logger utility: `logInfo()`, `logSuccess()`, `logError()`, `logWarn()`, `logDebug()`
- Use `startSpinner()` / `stopSpinner()` for long operations

## Commit Message Rules

- Write concise, descriptive commit messages
- Use imperative mood ("Add feature" not "Added feature")
- NEVER include repetitive compliance statements

## Common Pitfalls

- Don't run `quarto render` without MiKTeX on PATH
- Don't modify generated PDF files directly
- Don't forget to set `.env` before running generation
- Don't assume page count from Markdown length — always use the PDF for accurate counts
- Don't use `tsc` — use `tsgo` (the native TypeScript compiler)
